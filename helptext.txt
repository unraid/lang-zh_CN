**彩色状态指示器**彩色指示器在*阵列设备*中每行开头的意义如下：

<i class='fa fa-circle orb green-orb'></i>正常运行，设备处于活动状态。

<i class='fa fa-circle orb grey-orb'></i>设备处于待机模式（休眠）。

<i class='fa fa-warning orb yellow-orb'></i>设备内容已仿真。

<i class='fa fa-times orb red-orb'></i>设备已禁用，内容已仿真。

<i class='fa fa-square orb blue-orb'></i>新设备。

<i class='fa fa-square orb grey-orb'></i>设备不存在，位置为空。

**标识**是用于唯一识别存储设备的*签名*。签名
包括设备型号、序列号、Linux 设备 ID 和设备大小。

**温度**直接从设备读取。您可以在 [设置/显示首选项] (Settings/DisplaySettings)
页面配置要使用的单位。我们不读取休眠硬盘驱动器的温度，
因为这通常会使它们重新激活，所以改为显示 `*` 符号。我们还会显示 `*` SSD 和闪存设备的符号，
尽管有时这些设备确实报告了有效的温度，
但有时它们还是返回值 `0`。

**大小, 已用, 可用**报告总设备大小、已用空间和剩余空间。这些
还可以在 [显示首选项] (Settings/DisplaySettings) 页面上配置单位。
由于文件系统开销，空磁盘使用的空间量也不会为零。

*注意：对于多设备缓存池，此数据是 btrfs 返回的整个池的数据。*

**读取, 写入**是发送到设备 I/O 驱动程序的 I/O 请求的计数。这些统计数据可以
随时清除，请参阅下面的阵列状态部分。

**错误**用来对设备 I/O 驱动程序报告的*不可恢复*的错误计数。
由于出现不可恢复的阵列读取错误而导致的数据丢失，将使用奇偶校验重建功能即时填充（并且我们尝试将此数据写回发生故障的扇区）。
任何不可恢复的写入错误都会导致*禁用*磁盘。

**FS**指的是在设备的分区 1 中检测到的文件系统。

**查看**列中有一个文件夹图标，表示设备已*挂载*。单击此图标
浏览文件系统。

如果在 [设置/显示首选项] (Settings/DisplaySettings) 页面开启了 "显示阵列总计"，则
会包含一行**总计**，其中提供设备统计数据，包括设备的
平均温度。

必须停止阵列才能更改阵列设备分配。

UnRAID 阵列由一个或两个奇偶校验磁盘和多个数据磁盘组成。数据
磁盘专门用于存储用户数据，奇偶校验磁盘提供必要的冗余
以从磁盘故障中恢复。

由于数据不跨阵列条带化，因此奇偶校验磁盘必须与最大数据磁盘相同或比它更大。
奇偶校验磁盘也应当是您性能最高的驱动器。

每个数据磁盘都有自己的文件系统，并且可以
作为独立共享导出。

单击设备名称以配置各个设备设置，并启动某些实用程序。

**池设备**是单一设备，或多个设备的池，位于 Unraid 阵列的*外部*。可以像阵列设备一样将其导出以进行网络访问。在 Unraid 阵列之外可以显著加快写入访问速度。

有两种配置池设备的方法：

1. 作为单一设备，或
2. 作为多设备池。

当配置为单一设备时，您可以使用任何受支持的文件系统（btrfs、reiserfs、
或 xfs）格式化该设备。此配置可提供最高的性能，但代价是不提供数据保护 -
如果单一池设备发生故障，其中包含的所有数据都可能会丢失。

当配置为多设备池时，Unraid OS 将自动选择 *btrfs-raid1* 格式（同时用于数据
和元数据）。btrfs 允许将任意数量的设备添加到池中，并且保证
每个数据副本都可以写入两个不同的设备。因此该池可以承受单一磁盘故障，不会丢失数据。

启用［用户共享］(/Settings/ShareSettings) 时，可以将用户共享配置为
自动使用池设备，以加快写入速度。
可以计划一个称为 *mover* 的特殊后台进程
定期运行，以将用户共享文件从缓存移到阵列。

:array_devices_help:
#### 分配设备

一个 unRAID 磁盘阵列由一个奇偶校验磁盘和多个数据磁盘组成。数据磁盘
专门用于存储用户数据，而奇偶校验磁盘提供必要的冗余，
以从任何单一磁盘故障中恢复。

请注意，在引用阵列存储设备时，我们会慎用*磁盘*一词。
当引用实际硬盘驱动器 (HDD) 设备时，我们使用*硬盘驱动器*（有时也称为*驱动器*）一词。
这是因为在 RAID 系统中，可以读取/写入的阵列磁盘的相应
硬盘驱动器可能已禁用甚至已丢失! 另外，懂得询问 “将哪个设备
分配为奇偶校验磁盘？” 或 “哪个设备对应于 disk2 ？”都很有用。

因此我们需要一种方法来将硬盘驱动器分配给阵列磁盘。当阵列停止时，
可以在主页上完成此操作。每个阵列磁盘都有一个下拉框，其中列出了所有
未分配设备。要分配设备，只需从列表中选择它即可。每次进行设备
分配时，系统都会更新配置文件以记录分配。

#### 要求

与在所有阵列设备上对数据进行条带化的传统 RAID 系统不同，unRAID 服务器
会将文件存储在各个硬盘驱动器上。因此，所有文件写入操作都将涉及
所写入的数据磁盘和奇偶校验磁盘。鉴于这些原因，

*奇偶校验磁盘的大小必须大于或等于任何数据磁盘的大小，

以及

*在可选择的情况下，奇偶校验磁盘应当是集合中速度最快的磁盘。

#### 指导方针

在设计 unRAID 磁盘阵列时，应遵循以下步骤：

1. 确定要用于奇偶校验的硬盘驱动器，以及要用于
数据的硬盘驱动器（disk1 、disk2 等），并以某种方式进行标记。另外，找到每个硬盘驱动器的序列号并将其记在某处；日后需要使用此信息。

2. 安装硬盘驱动器设备，启动 unRAID 服务器和 webGui。如果这是全新的
系统构建，主页将显示未安装任何磁盘。这并不意味着系统无法检测
到硬盘；这只是意味着尚未分配任何磁盘。

3. 还记得在步骤 1 中记录的序列号吗？对于奇偶校验磁盘和每个数据磁盘，
请根据其序列号从下拉列表中选择合适的硬盘驱动器。

#### 热插拔

如果硬件支持，也可以将硬盘驱动器*热插拔*到服务器中。例如，
如果使用的是硬盘驱动器架，可以在通电且阵列已停止的情况下
将它们简单地插入服务器。刷新主页后，未分配的新设备将出现在分配
下拉列表中。

#### 后续步骤

分配所有硬盘驱动器后，请参考下面的阵列状态部分
并启动阵列。
:end

:encryption_help:
#### 加密输入

阵列停止后，用户可以指定一个新的加密密钥。请注意，当使用
特定密钥格式化设备后，只有使用相同的密钥才能将其打开。更改加密密钥需要
重新格式化已加密设备，**导致这些设备上所有的现有数据永久丢失。**

#### 密码

输入最多 512 个字符的密码。强烈建议仅使用 [ASCII 表](https://en.wikipedia.org/wiki/ASCII) 中前面 128 个字符中的 95 个可打印字符，因为它们始终具有相同的二进制表示形式。
根据系统配置，其他字符可能具有不同的编码，您的密码
将无法使用其他编码。如果您希望使用更长的密码或包含二进制数据，请改为上载密钥文件。

请参考 [cryptsetup FAQ]（https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects）
以了解构成*安全*密码的元素。

**牢记**这个密码。**如果丢失密码，已加密的内容将无法恢复！**

#### 密钥文件

选择具有存储的加密密钥的本地密钥文件或二进制文件。密钥文件的最大大小为 8M（8388608 字节）。

**备份**您的本地密钥文件.**如果丢失，已加密的内容将无法恢复！**
:end

:info_btrfs_balance_help:
**平衡**将运行 *btrfs balance* 程序以重新条带化所有池设备上的盘区，例如，将池从 raid1 转换为 raid0，或从 raid0 转换为 raid1。

当执行*完全平衡*时，基本上是将所有内容重写到当前文件系统中。

*模式转换*会影响 btrfs 数据范围；元数据始终使用 raid1，并在必要时通过任何平衡操作转换为 raid1。

运行时间可能很长，具体取决于文件系统大小和设备速度。

在创建多设备池时，Unraid 使用以下默认选项：

`-dconvert=raid1 -mconvert=raid1`

有关更完整的文档，请参阅 btrfs-balance [管理](https://btrfs.wiki.kernel.org/index.php/Manpage/btrfs-balance)

*注意：raid5 和 raid6 一般仍被 Linux 社区视为**实验性***
:end

:share_edit_cache_pool_help:
指定是否可以将共享中写入的新文件和目录写入到缓存磁盘/池中。
此设置还会影响 *Mover* 行为。

**否**将禁止新文件和子目录写入缓存磁盘/池。
*Mover* 将不会执行任何操作，因此该共享的所有现有文件都将保留在缓存中。

**是**表示应将所有新文件和子目录写入缓存磁盘/池，前提是
缓存磁盘/池中存在足够的可用空间。
如果缓存磁盘/池上没有足够的空间，则会在阵列上创建新的文件和目录。
当调用 *Mover* 时，文件和子目录会从缓存磁盘/池转移到阵列上。

**仅于**表示必须将所有新文件和子目录写入高速缓存磁盘/池。
如果缓存磁盘/池上的可用空间不足，*创建*操作将失败，并显示*空间不足*状态。
*Mover* 将不会执行任何操作，因此阵列上此共享的所有现有文件都保留在那里。

**首选**表示应将所有新文件和子目录写入缓存磁盘/池，前提是
缓存磁盘/池中存在足够的可用空间。
如果缓存磁盘/池上的可用空间不足，则会在阵列上创建新的文件和目录。
当调用 *Mover* 时，文件和子目录从阵列转移到缓存磁盘/池。

**注意：**Mover 决不会移动当前正在使用的任何文件。
这意味着如果要移动与 Docker 或虚拟机等
系统服务关联的文件，则需要在 Mover 运行时禁用这些服务。
:end

:smb_case_sensitive_names_help:
控制文件名是否区分大小写。

使用默认设置**自动**时，支持区分大小写文件名 (Linux CIFSVFS) 的客户端
可基于数据包告知 Samba 服务器，它们希望以区分大小写的方式
访问文件系统（以支持 UNIX 区分大小写的语义）。Windows 系统不支持区分大小写的文件名，因此
将该选项设置为**自动**的效果，与将其设置为“否”相同；但是，会保留 Windows 客户端传递的文件名的大小写。对于大型目录，此设置可能会导致
性能降低，因为 Samba 必须进行文件名搜索并匹配传递的名称。

设置为**是**表示在客户端通过的情况下创建文件，并且只能使用相同的大小写进行访问。
这将加快大型目录的访问速度，但是使用此设置可能会令某些 Windows 应用程序无法正常运行。
例如，如果创建了 'MyFile'，但某个 Windows 应用程序尝试打开 'MYFILE' （Windows 允许这种情况），会找不到该文件。

**强制小写**的值很特殊：所有传入客户端文件名（不仅是新文件名）都将设置为小写。
换句话说，无论在 Windows 端创建了何种大小写混用的名称，都将以全小写来存储和访问该名称。
这样可以确保所有 Windows 应用程序正确查找无论大小写如何的任何文件，但大小写不会保留在文件夹列表中。
请注意，仅应为新共享配置此设置。
:end

:mgmt_certificate_expiration_help:
**预配**可用于分配来自于 [Let's Encrypt](https://letsencrypt.org/) 的*免费* SSL 证书，
并随后上传到服务器。备注：**强烈**建议在此情况下使用静态 IP 地址。

**更新 DNS** 可用于手动启动 unraid.net 上服务器 FQDN 的 DNS A 记录更新。注意，DNS 传播更改可能需要 1 分钟到几小时不等（我们设置的 TTL 为 60 秒）。

备注：如果您的路由器或上游 DNS 服务器启用了 
[DNS 重新绑定保护] (https://en.wikipedia.org/wiki/DNS_rebinding)，**预配**可能会失败。
DNS 重新绑定保护可防止 DNS 解析到专用 IP 网络范围。在有漏洞/不安全 "web" 接口的传统设备所在的本地 LAN 中，DNS 重新绑定保护将作为一项安全特性。

DNS 重新绑定保护的来源之一可能是您的 ISP DNS 服务商。这种情况下可以切换到其他DNS 服务商来解决问题，例如切换到谷歌的公用 DNS。

而更普遍的情况下，DNS 重新绑定保护可能是在您的路由器中启用的。大多数消费级路由器不执行 DNS 重新绑定保护；
但如果执行了保护，应当有一条配置设置用于关闭此功能。

然而，较高端的路由器往往会**启用**DNS 重新绑定保护。通常可以通过一些途径将其完全关闭，或是根据域名有选择性地关闭。例如：

**DD-WRT:** 如果您正在使用启用了 DNS 重新绑定保护的 "dnsmasq"，可以在路由器的配置文件中添加下面这一行：

`rebind-domain-ok=/unraid.net/`

**pfSense:**如果您正在使用 pfSense 内部 DNS 解析器服务，可以添加以下自定义选项行：

`server:`<br>
`private-domain: "unraid.net"`

**Ubiquiti USG router:**您可以添加以下配置行：

`set service dns forwarding options rebind-domain-ok=/unraid.net/`

**OpenDNS:** 转到“设置 -> 安全”，然后*移除*位于“可疑回应   
 - 阻止内部 IP 地址”旁的复选框。这是一个全有或全无的设置。
 
若以上方法均告失败，可以在电脑的 *host* 文件中创建一个条目，以覆盖外部 DNS，
并直接将您服务器的 unraid.net FQDN 解析到它的本地 IP 地址。
:end

:nfs_tunable_fuse_remember_help:
当启用 NFS 时，这一可调选项可用于减轻或解决您可能在 NFS 客户端上遇到的
"NFS Stale File Handles" 实例。

本质上来说，(fuse_remember) 会告知内部子系统（名为 "fuse" ）“记住”或“缓存”
与用户共享关联的文件和目录信息多长时间。当 NFS 客户端尝试访问服务器上的
文件（或目录），且该文件（或目录）没有被缓存时，可能会遇到
"stale file handle"。

这一可调选项的数值是缓存文件/目录名条目的秒数，
默认值为 330，也就是 5.5 分钟。有两个可以设置的
特殊值：

*0 意味着完全不缓存文件/目录名，以及
*-1 意味着总是缓存文件/目录名（或者直到阵列停止）

如果您启用了 NFS，但只计划将它用于磁盘共享，不用于用户共享时，可以设为 0 值。

占用大约 108 字节的速度增长。是否会导致内存不足，取决于
服务器安装了多少内存以及您通过 NFS 访问了多少文件/目录。
:end

绑定是一种将多个物理以太网接口合并为单独一个*绑定*接口（名称为 **bond0**）的特性。
该选项可用于提高连接冗余和/或系统的吞吐量。

:eth_bonding_mode_help:
**模式 0（平衡 - rr）**<br>
该模式将按顺序传输数据包，从第一个可用的从属接口一直传输到最后一个接口。
如果两个实际接口是该绑定中的从属接口，并且有两个数据包到达绑定接口之外的位置，则第一个数据包在第一个从属接口中传输，第二个数据包在第二个从属接口中传输。
第三个数据包在第一个从属接口中发送，依此类推。
该模式提供负载平衡和容错。

**模式 1（主从模式） - 默认**<br>
该模式将其中的一个接口置于备份状态，并且只在主接口丢失链路时激活该接口。
在某个时间实例内，该绑定中只有一个设置的接口处于活动状态。只有在设置的主接口出现故障时，才会激活设置的备份接口。
该模式提供容错。

**模式 2（平衡 - 异或）**<br>
该模式基于异或公式传输数据包。源 MAC 地址与目标 MAC 地址通过异或公式对从属接口数取模。(类似于ip hash, 只不过是用mac地址计算)
每个目标 MAC 地址将选择相同的从属接口，从而提供负载平衡和容错能力。

**模式 3（广播）**<br>
该模式在所有从属接口中传输所有数据包。该模式用得最少（仅用于特殊用途），且只提供容错。

**模式 4（802.3ad）**<br>
该模式被称作*动态链路聚合*。它会创建聚合群组，共享相同的速度和双工设置。
它需要支持 IEEE 802.3ad 动态链路的交换机。
根据传输哈希策略对传出流量的从属接口进行选择，可以通过 xmit_hash_policy 选项来从默认的简单异或策略更改为其他策略。
注意，并非所有的传输策略都符合 802.3ad 的要求，尤其是 802.3ad 标准中 43.2.4 章节关于数据包错序的要求。
不同的对等实现对于不合规有着不同的容限。

**模式 5（平衡 - tlb）**<br>
该模式被称为*自适应传输负载均衡*。传出流量根据每个从属接口的当前负载和队列进行分发。
传入流量由当前从属接口接收。

**模式 6（平衡 - alb）**<br>
该模式被称为*自适应负载均衡*。它包括平衡 - tlb 和面向 IPV4 流量的接收负载平衡 (rlb)。
接收负载均衡利用 ARP 协商来实现。
绑定的驱动程序将拦截服务器发出的 ARP 回应，并且用该绑定中某个从属接口的唯一硬件地址重写源硬件地址，
以便不同的客户端对该服务器使用不同的硬件地址。

*建议设置为模式 1（主动备份）。其他模式虽然允许您设置特定的环境，但可能需要适当的交换机支持。
选择不支持的模式可能导致通信中断。*
:end

:apc_battery_time_help:
如果在电源故障期间，UPS 依靠电池运行*超时*数秒或更长时间；apcupsd 将启动系统关闭。零值将禁用此计时器。

如果您有 Smart UPS，您最可能希望通过将此计时器设置为零来禁用它。
这样，您的 UPS 将继续使用电池，直到剩余电量百分比降至或低于*电池电量*或剩余电池运行时间降至或低于*分钟*值。

当然 - 在测试时 - 如果您拔下电源插头，将其设置为 60 会导致系统快速关闭。
如果您使用的是旧式 UPS，则将其设置为小于您可以使用电池供电的时间。
:end
